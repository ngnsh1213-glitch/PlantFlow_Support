# 무탭 부재 식별 기반 치수·앵커 (cycle 90 = 단계 B)

## 1. 왜 이 사이클이 필요한가

RC1→RC2→RC3에서 **비율 추정으로 앵커·치수를 맞추려는 시도가 3회 연속 실패**했다.
RC1에 맞추면 RC2가 깨지고, RC2에 맞추면 RC3이 깨진다. 근본 원인은 하나다.

> 부재를 식별하지 않고 **서포트 전체 바운딩 박스(`supportPaperExt`)와 비율 상수**로
> 치수 구간과 콜아웃 앵커를 추정하고 있다.

**추정 조정을 중단하고 실측 박스 기반으로 전환한다.**

## 2. 실측 근거 (2026-07-20 라이브 로그)

RC2 `member-spike`:
```
i=0  wcsDims=(490,180,500)   paperBox=(286.5,268.9)~(384.5,368.9)
i=1  wcsDims=(168,50,148)    paperBox=(327.7,359.5)~(361.3,389.1)   = 파이프 클램프
i=2  wcsDims=(89.1,480,89.1) paperBox=(335.59,368.9)~(353.41,386.72) = 세로 MEMBER
```
- **세로 MEMBER 실제 x = 335.6~353.4.** 현행 `vertical` 앵커 추정식은 `minX + barPaperH*0.5 = 293`
  → **40 이상 빗나감**. 재추출해도 허공.
- **가로 치수 490의 정체 = 하부 베이스 플레이트 좌측 끝**(사용자 확인).
  실제 가로 MEMBER는 450. `supportPaperExt`가 플레이트를 포함해 40이 과대계상된다.
  `split=(290,200)`의 좌측 290이 250이어야 하는 이유.

## 3. 설계

### 3-1. `member-spike`를 식별 헬퍼로 승격
현재 `NotabMemberGeometrySpike()`는 로그 전용이다. 다음을 반환하는 헬퍼로 바꾼다.
```
List<NotabMemberBox> { ObjectId Id; string Handle; Extents3d PaperBox; Vector3d WcsDims; double Aspect; }
```
- 투영은 기존 `NotabProjectWcsToPaper()` 그대로 재사용(신규 계산 없음).
- **로그는 유지**한다(회귀 진단 자산).

### 3-2. 부재 분류
관측된 구분 신호:
- **세로 MEMBER**: 페이퍼 박스가 세로로 길다(RC2/RC3에서 폭 17.82, 종횡비 5.39).
- **클램프/부속**: 파이프 중심에 근접(`pipeDist` 최소), 크기 작음.
- **가로 MEMBER + 베이스 플레이트**: 하나의 솔리드(i=0)로 묶여 있을 가능성.
  → **이것이 미해결 지점.** 490(플레이트 포함)에서 450(가로재)을 분리해야 한다.

**분류는 임계값 하드코딩이 아니라 규칙으로** 세우고, 분류 결과를 로그에 남겨
타입별로 검증 가능하게 한다. 분류 실패 시 **기존 동작으로 폴백**하고 명시 로그(무음 실패 금지).

### 3-3. 콜아웃 앵커를 실측 박스에서 산출
- 세로 MEMBER 콜아웃 앵커 = 세로 MEMBER `PaperBox`의 몸통 중앙.
- 가로 MEMBER 콜아웃 앵커 = 가로 MEMBER `PaperBox` 기준.
- **비율 상수(`MemberAnchorSide` vertical/top/bottom 추정식)는 폐기 또는 폴백 전용으로 강등.**
- 좌우는 cycle 88의 R1~R4를 그대로 따른다(규칙 변경 없음).

### 3-4. 치수 값은 기하가 아니라 **SupportParams**에서 (자문 반영, 계획 정정)

**정정**: "가로 MEMBER 박스 폭 = 450"은 **틀렸다**. 가로재와 베이스 플레이트가 한 솔리드라
그 bbox는 여전히 490이다. 450은 기하에서 나오지 않는다.

라이브 `support params dump` 실측으로 규칙이 확정됐다:
```
RC1  A=350 A1=100          → A+A1 = 450   (좌 350 / 우 100)
RC2  A=400 A1=50  A2=200   → A+A1 = 450   (좌 250 / 우 200)
RC3  A=250 A1=250 A2=250   → A+A1 = 500   (좌 250 / 우 250)
```
- **총 폭 = `A + A1`**
- **우측 분할 = `A2`**(없으면 `A1`), **좌측 = 총 폭 − 우측**
- 세 타입 모두 사용자 기대치와 일치. 490은 파라미터 어디에도 없다(= 플레이트 bbox).

적용 범위 분리(자문 지적):
- **치수 값**과 **분할** = SupportParams.
- **보조선 위치** = 부재 우측 끝을 기준으로 총 폭만큼 좌측. 그대로 bbox를 쓰면
  숫자만 450이고 보조선은 490 폭에 남는 불일치가 생긴다.
- 파라미터 누락·비유한·0 이하면 **기존 `s_isoRealWidth` 경로로 폴백** + 사유 로그.
- 세로 치수는 `param(F2)`가 이미 정상 동작 중(RC2 raw=500, RC3 raw=600 확인). **건드리지 않는다.**

### 3-5. 치수 오프셋
현행 `offset = txt*1.5`(=12)는 RC3에서 클램프 위를 지나 부족하다.
- 단기: 기본값 상향(라이브로 적정값 확정).
- 지향: 서포트 bbox가 아니라 **실제 그려진 내용의 바깥 끝**에서 띄우기.
  단, 이번 사이클에서 무리하게 묶지 말고 오프셋은 **상수 상향까지만** 한다.

## 4. 검증
1. RC1/RC2/RC3 추출.
2. 가로 치수 = RC2 450(250/200), RC1 450, RC3 500 — 베이스 플레이트 폭이 아닌 부재 폭.
3. L 부재 콜아웃 화살표가 **세로 MEMBER 몸통**에 닿음(허공 0).
4. 치수선이 클램프·부재와 겹치지 않음.
5. **GD1/GD2/GD3 회귀 없음** — 부재 식별 도입이 GD 경로를 깨지 않아야 한다.
   분류 실패 시 폴백으로 기존 동작 유지.

## 5. 하지 말 것
- 비율 상수 추가 튜닝(이번 사이클의 존재 이유가 그 폐기다).
- cycle 88 좌우 규칙 R1~R4 변경.
- 세로 치수 `param(F2)` 로직 변경(정상 동작 중).
