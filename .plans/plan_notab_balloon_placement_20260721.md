# 밸룬 배치 밀집 해소 (cycle 96)

## 1. 증상
밸룬 3개(F1/F2/P1)가 앵커에서 멀리 떨어진 한 영역으로 몰린다.
사용자 판정: "이런식으로 밀집되는 현상도 발생하네" (RC1~RC3 전 타입).

## 2. 시도와 실패 이력 (같은 문제에 3회 실패 — 추정 중단 근거)

| 시도 | 조치 | 결과 |
|---|---|---|
| 1 | 앵커를 부재 끝(포트) → **부재 중앙**으로 이동 | 의미는 맞았으나 사방이 막혀 **r 119~154로 급증** |
| 2 | 서포트 본체의 **리더 검사 면제** | **수치 무변화**. 계측 결과 차단자는 서포트가 아니라 치수(`extLeaderBy{unnamed}`) |
| 3 | 장애물 패딩 8→2 분리 + 밸룬 최소이격 40→20 | **개선 없음. 일부 악화**(F2 61.8→71.8, P1 19.6→39.6) |

## 3. 실측 (2026-07-21 08:03~08:04, 최신 바이너리)
```
RC2 F1 anchor=(339.5,368.9) center=(369.1,239.6) r=129.3 cost=48.79 scanned=181
       reject(oob/box/extLeader/calloutLeader)=1852/659/630/974  extLeaderBy{unnamed=630}
RC2 F2 anchor=(311,318.9)   center=(265.5,381.08) r=71.8 cost=47.30 scanned=1983
RC2 P1 anchor=(298,269.5)   center=(260.4,297.5)  r=39.6 cost=36.10
RC3 F1 r=149.3 cost=43.99 scanned=169   RC3 F2 r=134.3 cost=95.34 scanned=37
```

### ★유력 원인 — 비용 함수에서 거리가 사실상 공짜다
```csharp
cost = (costRefExt 밖으로 삐져나온 양 4방향 합)
     + radius * 0.01          // ← 거리 가중치
     + |fanOffset| * angleW;  // angleW 기본 0.0 (cycle88에서 0으로 낮춤)
```
- `radius * 0.01`이면 r이 100 늘어도 비용은 **1**만 는다.
- 반면 `costRefExt`(서포트+앵커 bbox) 밖으로 나가면 **삐져나온 거리만큼 그대로** 가산된다.
- 결과: **앵커에서 멀어도 costRef 안쪽에 머무는 후보가 최저 비용**이 된다.
  실제로 RC3 F1은 r=149인데 cost=43.99로, r이 작은 후보들보다 저렴하다.
- 텍스트 콜아웃은 어차피 바깥으로 밀어내는 성격이라 이 설계가 통했지만,
  **밸룬은 부재 옆에 붙어야** 의미가 있어 전제가 다르다.

### 보조 원인
- **치수가 리더를 막는다**: `extLeaderBy{unnamed=630/596}`. 치수가 서포트를 둘러싸
  부재 중앙에서 출발한 리더의 탈출로가 좁다.
- **기존 콜아웃 리더 교차**: `calloutLeader=974/964/1515`. 밸룬을 마지막에 배치하므로
  파이프·부재·U-bolt 리더가 이미 등록돼 있다.
- **후보 고갈**: RC3 F2는 `scanned=37`까지 떨어진다.

## 4. 자문에 부칠 쟁점
1. **비용 함수** — 밸룬에 거리 가중치를 크게(예: `radius * 0.5~1.0`) 주는 것이 맞는가.
   아니면 밸룬 전용 비용식을 따로 두는가.
2. **배치기 재사용 타당성** — `NotabCalloutPlacer`는 텍스트 상자 + 3정점 리더 전제다.
   밸룬(작은 원 + 직선 리더)에 사각 근사·부채꼴 탐색·좌우 하드 제약을 그대로 적용하는 것이
   과한가. 별도 경량 배치가 나은가.
3. **앵커 위치** — 부재 중앙은 표기 의미상 옳지만 탈출로가 없다.
   중앙과 끝의 절충(길이 25% 지점) 또는 부재 바깥쪽 가장자리가 나은가.
4. **치수 리더 완화** — 밸룬에 한해 치수 장애물의 리더 검사도 면제할지.
   (문자 상자 겹침은 유지)
5. **좌우 하드 제약(R1~R4)이 밸룬에도 적용되어야 하는가** — 텍스트 콜아웃은 좌우가
   판독성에 직결되지만, 밸룬은 부재 바로 옆이 더 중요할 수 있다.

## 4-A. 자문(Codex) 결론 — 전용 경로로 분리

**가설 확인**: `costRefExt`는 밸룬별 앵커가 아니라 `AppendNotabPaperDimensions()`에서
**한 번** 설정된다(서포트 + 파이프 중심 bbox). 밸룬 앵커는 비용 기준에 **포함되지 않는다**.
`preferDown=false`, `angleW=0`이므로 실질 비용은
```
cost = costRefExt 밖으로 나간 양 + radius × 0.01
```
RC2 F1의 `r=129.3, cost=48.79`는 반경항 ≈1.29 + 기준상자 이탈 ≈47.5의 조합이다.
**더 가까운 후보가 기준 상자 밖으로 조금만 더 나가면 진다.**

**★거리 가중치만 올리는 것은 네 번째 튜닝 실패가 될 가능성이 높다**(자문 명시).
`TryPlace`가 텍스트 콜아웃 전제를 다섯 개나 갖고 있기 때문이다:
좌우 한쪽 강제 / `minDx` 강제 / 외접 사각형 충돌 / 앵커→꺾임→상자 2선분 리더 모델 /
서포트 bbox 안쪽 선호.

### 권장 = 밸룬 전용 경량 배치
```
후보 중심 = anchor + 방향벡터 × 거리   (작은 거리부터 확장, 첫 허용에서 최저 비용 채택)
비용 = 거리 × 높은 가중치
     + 원·보조문자 bbox 겹침(하드 제약 또는 큰 페널티)
     + 리더 교차 페널티
     + 도면 밖 페널티
costRefExt 이탈 항은 밸룬에서 제거하거나 아주 약하게
```

### 제약 완화 우선순위 (밸룬에만, 텍스트 R1~R4는 보존)
1. 좌우 강제 → **소프트 선호**로 변경
2. `minDx` **제거 또는 반지름 수준**으로 축소
3. **360° 방향 탐색**, 치수 영역 쪽만 약한 페널티
4. 텍스트 콜아웃 R1~R4는 **그대로**

### 앵커 — "중심 유지 + 외곽 접점에서 출발"
부재 중앙은 표기 의미상 옳다. 다만 중앙은 부재·치수·기존 리더에 둘러싸이기 쉽다.
```
의미 앵커  = 부재 중심
리더 시작점 = 밸룬 방향을 향한 부재 외곽 접점
```
이러면 지시 의미는 유지하면서 리더가 부재 내부·치수 통로를 길게 가로지르지 않는다.
기존 `BoxEdgeToward()`가 유사 계산을 이미 갖고 있다.

### 치수 리더 완화 — 범위를 좁게
밸룬 **전용**, **리더 선분 검사만**, **치수 장애물만** 면제. 원·보조문자 겹침은 계속 금지.
위험: 치수선·보조선·문자 위를 리더가 지나 판독성이 떨어질 수 있다. 현재는 치수 전체
`GeometricExtents`를 사각형 하나로 다뤄 "문자는 피하고 선만 통과"를 구분하지 못한다.
후속 개선안 = 치수의 문자 영역과 선분 영역 분리(문자 하드 금지 / 선분 낮은 페널티).

### 추가 위험 (자문)
- 밸룬을 **외접 사각형**으로 충돌 판정해 원의 빈 모서리까지 점유로 본다 → 후보 과다 탈락.
- `calloutLeader=974`가 최대 거절원이다. **밸룬을 마지막에 배치하는 정책의 필연적 결과**이므로
  밸룬은 짧은 리더 우선 정책이 반드시 필요하다.
- tier 2는 리더 검사를 완전히 끈다. 진단에 **`tier`를 반드시 함께** 확인할 것.
- `TryPlace`의 반환 `center`는 실제 원 중심이 아니라 텍스트 상자 경계 중앙이다.
  전용 배치기로 분리하지 않으면 유지보수 시 재오해 위험이 크다.
- 번호(`F1`) MText 실제 폭이 반지름보다 넓을 수 있다. **최소 반지름을 문자 폭으로 검증**할 것.
